Maximizing Driver Earnings by Selecting the Best Configuration and Time to Start

A Simulation-Driven Optimization of Ride-Hail Policies Using Tree-Based Interactions

Once have a policy to decide if a trip should be taken or rejected then it’s important to validate if any of the next factors could cause an increase of decrease on taxi results after blocking the company for taxi driver:

  • Start zone
  • Days of the week
  • Hour of the day

Defining combinations to simulate

As we have 196 zones to evaluete, 3 days to evaluete for each day of the week (7), 2 taxi companies and we divide the day in 8 moments to measure (one each 3 hours), we end with an experiment of 65,856 days to simulate.

Running simulation

As I knew that running the hole process would take me several days I decided to store the results of simulations on disk for each zone. In this way I was able to keep the experiment as I was expecting and stop the process between days to allow the computer to rest between days and redice the risk of loossing all the progress I had problems with memory while using the computer for dealy activities.

To avoid the overhead of using a IDE the next code was source from terminal after connecting with ssh NycTaxi to consume the fewer results as possible.

Note

In the below code you see an assigning the starting zone to DOLocationID as the simulation uses the last zone of the configuration trip (the one we are creting) to starting picking trips from that zone.

consolidate_all_simulations <- function(zone_id_data, board, sim_start_days) {

Finding hidden patterns

Based on the prior exploration it’s clear that the most important rule found so far is to take Uber as the company, but to improve earning we need to find better interactions between variables and for that reason we are going to start with tree model to see the model can find the most important interactions to explain changes in the target variable.

To tune the model we know that the must important hyperparam is the tree_depth to limit how many splits the tree can make, which controls vertical complexity of the model and as result we want to use the minimum value posible.

As we have a lot of data to explore due the number of zones of the experiment we know that the min_n to set the minimum number of data points required in a node to allow a further split won’t be a important hyperparam as must of the split will have many data points to avoid overfit.

DecisionTreeSpec <- decision_tree(
  mode = "classification",
  tree_depth = tune("tree_depth"),
  min_n = 10
)

To help the tree finding the important pattern we are going to apply the following transformations to the orininal data:

  • As start_zone present high cardinality of zone we substitute with hourly_wage_mean_by_zone after isolating the effect of taxi company.
  • To allow the model to see hour of the day and the time of the week as the cycles they represent in the real live where the same time is repited very 24 hours and the same time in the week is repeted every 168 hours we are creating the variabes day_cycle_sin_1 and day_cycle_cos_1 instead of just showing time_hour and week_cycle_sin_1 and week_cycle_cos_1 instead of using the original week_day.
  • We are showing the taxi_company company as boolean variable to keep all variable numeric.
DecisionTreeRecipe <- recipe(
  is_top_trip ~ taxi_company +
    start_zone +
    week_day +
    time_hour +
    simulation_id +
    daily_hourly_wage_mean,
  data = InitialZoneSimResults
) |>
  # Creating variables to use later
  step_mutate(
    day_cycle = time_hour,
    week_cycle = (week_day - 1) * 24 + time_hour,
    hourly_wage_mean_by_zone = paste0(start_zone, "-", taxi_company)
  ) |>
  # Changing zones with their mean
  step_lencode(
    hourly_wage_mean_by_zone,
    outcome = vars(daily_hourly_wage_mean),
    smooth = FALSE
  ) |>
  # Transforming company to binary variable
  step_dummy(taxi_company) |>
  # Creating Cycles
  step_harmonic(
    day_cycle,
    cycle_size = 24,
    frequency = 1,
    keep_original_cols = FALSE
  ) |>
  step_harmonic(
    week_cycle,
    cycle_size = 168,
    frequency = 1,
    keep_original_cols = FALSE
  ) |>
  # Saving original vars
  update_role(
    simulation_id,
    daily_hourly_wage_mean,
    start_zone,
    time_hour,
    week_day,
    new_role = "additional info"
  ) |>
  # Removing original variables
  step_upsample(is_top_trip, over_ratio = tune("over_ratio"))

Now after all this configuration we are ready to consolidate all that logic into a compact workflow that will help us to fit and evaluate our model using all tools from tidymodels.

DecisionTreeWf <-
  workflow() |>
  add_recipe(DecisionTreeRecipe) |>
  add_model(DecisionTreeSpec)

As this as simple model and we just need to tune 2 params we can use the grid method the find the best trip possible with 10 fold cross validation to estimate the model error without taking testing data, as we just need to understand the results of the experiment rather than using this model to make predictions.

DecisionTreeWfRegularGrid <- grid_regular(
  tree_depth(range = c(2L, 16L)),
  over_ratio(range = c(0.1, 1)),
  levels = c(tree_depth = 5, over_ratio = 10)
)

set.seed(5878)
InitialZoneSimResultsResamples <- vfold_cv(
  InitialZoneSimResults,
  v = 10,
  strata = is_top_trip
)

DecisionTreeWfTuned <- tune_grid(
  object = DecisionTreeWf,
  resamples = InitialZoneSimResultsResamples,
  grid = DecisionTreeWfRegularGrid,
  metrics = metric_set(recall, precision, brier_class),
  control = control_grid(
    verbose = TRUE,
    parallel_over = NULL,
    save_pred = FALSE
  )
)

pin_write(
  BoardLocal,
  DecisionTreeWfTuned,
  name = "DecisionTreeWfTuned",
  type = "qs2"
)

plot_custom_conf_matrix <- function(dt) {

    dt |>

    summarize(Freq = n(), mean_pred_yes = mean(.pred_yes)) |>
    mutate(truth_prop = Freq / sum(Freq)) |>
    ungroup()

    procced_data |>
    filter(Truth == "yes", Prediction == "yes") |>
    pull(truth_prop) |>
    percent()

With 65,856 simulated days all coefficients are statistically significant, so the focus should be on the magnitude of each effect.

The regression reveals a clear hierarchy of factors:

  • A driver that use Uber can earn up to $8.16 more per hour compared to a driver that use Lyft.
  • Starting at a favorable time of day can add up to $5.08 per hour (captured by day_cycle_cos_1), while an unfavorable hour can subtract up to $2.94.
  • Day-of-week effects are small (under $1.58 in either direction), confirming that what time of day the driver starts matters far more than which day of the week they work.
  • A driver who follows the tree model’s accept/reject strategy earns an additional $5.04 per hour on top of all the above. This reflects the specific company-and-timing combinations the tree identified as high-value that no single variable captures independently.

In practice, a driver making all the right decisions — starting with Uber at a favorable time and following the tree model’s strategy — can move from the baseline of $54/hour to approximately $70/hour, representing a 34% improvement in hourly earnings.
